# M1 — the interpret step over-credits nominal/at-spec checks as POSITIVE support

## Audit (the live mechanism)

**Confirmed real, not a scorer artifact.** On `case5` (`no_clean_cause`, gold = ABSTAIN) the
scorer keys accuracy / abstention_calibration on `answer.answer_type == "abstain"`; the live run
(`cap_case5.json`) returned `answer_type=="cause"`, `root_cause=="part.laser_module"` — a genuine
wrong conclusion, scored accuracy 0 / abstention_calibration 0.

**The mechanism, read off the live Investigation Graph (`cap_case5.json`).** Every check on case5
reads nominal/at-spec, yet `interpret_result` credited the *rule-out* checks **positively** to the
laser hypothesis (H4 / `part.laser_module`):

| live evidence | check | what it actually showed | over-credited link (the bug) |
|---|---|---|---|
| E1 | `onset_vs_event_check` | onset predates reboot → reboot is non-causal (a DEMOTION) | `+1.0` to H3 (tec), H4 (laser), H5 (detector) |
| E4 | `spatial_intensity_check` | `localized=false` → only RULES OUT window | `+1.5` to H4, `+0.5` to H2/H3/H5 |
| E3 | `detector_health_check` | `bias_drift=false`, fully nominal | `+0.5` to H4 (laser) "by elimination" |
| E2 | `tec_load_check` | nominal, headroom | (correctly all negative) |

Those bogus positives summed laser to `log_odds = +2.0` (conf **0.881**, margin **0.612**) — a
**confident, DOMINANT, wrong** leader. The margin-aware abstain gate (`loop.py:475`,
`abstain = conf <= tau_min or margin <= tau_margin`) is structurally unable to catch this: margin
0.612 ≫ τ_margin. The prior investigation correctly flagged this as the deferred **M1** item and
noted the gate was "correct but insufficient" for case5. The fix must be at the interpret/evidence
layer, where the bogus positive links are minted.

**The constraint, read off `cap_case2.json`** (`laser_aging`, gold = `part.laser_module`, currently
PASSES live): laser concludes via the SAME over-crediting — `spatial_intensity_check localized=false`
gave `+1.5` to laser plus elimination from the nominal detector check. There is **no affirmative
check in the existing menu that points at the laser**: `temp_correlation` (r=-0.01), `tec_load`
(48%, in spec), `detector_health` (nominal) all read False. So a naive "nominal checks add no
positive links" fix strips laser's only support and makes case2 abstain too. Telemetry confirms the
real difference: case2 `laser_power_mW` is sustained **-11.2%**, case5 is flat **+0.2%** (a transient
that recovered).

## Fix (the diff + #5 justification)

Two coupled changes (`investigation/patches/m1.diff`):

**1. Affirmative-evidence gate (the M1 fix).** Each check in `checks.py` now carries a general
boolean `affirmative` = "this check found a positive anomaly consistent with some fault", computed
from the check's own structural outcome:

| check | `affirmative` predicate |
|---|---|
| `config_diff` | `any_change` |
| `spatial_intensity_check` | `localized` |
| `temp_correlation_check` | `correlated` |
| `tec_load_check` | `at_limit or losing_setpoint` |
| `channel_sanity_check` | `stuck` |
| `detector_health_check` | `bias_drift` |
| `common_mode_check` | `common_mode` |
| `onset_vs_event_check` | `not onset_predates_event` (onset ALIGNS) |
| `laser_power_check` (new) | `low_power` (sustained decline) |

In `interpret_result` (`llm.py`), for a `run_check` action whose result is **not** affirmative,
positive (`+`) links are dropped; negative (rule-out) links are kept. A nominal check can rule out,
never rule in. A missing/None `affirmative` (non-check action) is treated as affirmative, so the gate
never strips legitimate evidence it can't reason about.

**2. Give laser an affirmative signal (`laser_power_check`).** New check in `checks.py`, wired into
`environment.py` (`_CHECKS` + `_run_laser_power_check`, auto-listed in the action menu via
`list_checks()`) and into the scorer's `CHECK_IDS` (`bespoke.py`). It measures emitter power directly
— recent mean vs baseline mean — and is affirmative iff `recent/baseline <= 0.97` (a ≥3% **sustained**
decline; a transient dip that recovered leaves recent≈baseline → not affirmative). Threshold chosen
from the real data: across the 8 cases the ratio is **≤0.945** on every genuine power-decline case and
**≥0.996** on every flat case — a wide, unambiguous gap (nearest pair 0.9446 vs 0.9966).

**#5 justification (structural, verified across all 8 cases).** The gate encodes a domain-general
abduction principle — *only an affirmative anomaly affirms a hypothesis; nominal evidence only rules
out* — and every `affirmative` predicate is a per-check structural fact (the check's own anomaly
condition), never a per-case constant or name. Verified live across all 8 cases (run via the env):

| case | gold cause | affirmative check(s) that fire | gold has an affirmative path? |
|---|---|---|---|
| case1 tec_degradation | part.tec | temp_correlation, tec_load(losing_setpoint), channel_sanity, **laser_power** | ✓ |
| case2 laser_aging | part.laser_module | common_mode, **laser_power** | ✓ (laser_power is the load-bearing one) |
| case3 window_contam | part.window | spatial_intensity(localized) | ✓ |
| case4 calibration_drift | sub.calibration | config_diff(any_change) | ✓ |
| **case5 no_clean_cause** | **ABSTAIN** | **none** | **✓ — no affirmative check fires → abstains** |
| case6 detector_bias | part.detector | detector_health(bias_drift) | ✓ |
| case7 tec_variant | part.tec | tec_load(losing_setpoint), **laser_power** | ✓ |
| case8 common_mode | sub.power | config_diff(any_change), common_mode, detector_health, **laser_power** | ✓ |

Every gold cause keeps an affirmative path; the abstain case is the *only* one with none. No case is
keyed by name. (`laser_power_check` is affirmative on case1/2/7/8 — wherever power genuinely declined,
which includes TEC and common-mode power faults — and the LLM decides which hypotheses it bears on; it
is a structural "source-side power loss" signal, not a laser-exclusive flag, which is why it is
general rather than a case2 patch.)

## Fix audit (regression risk)

- **Does the LLM have to RUN `laser_power_check` for case2 to conclude? Yes — and that is the residual
  exposure.** Post-M1, the nominal checks no longer lend laser any positive support, so case2's
  conclusion now depends on the affirmative `laser_power_check` actually being proposed and run. If
  the live LLM never selects it, laser has no affirmative path and case2 would **abstain** (a false
  abstention — a *new* M1-adjacent risk created by the fix). Mitigations in place: the check is in the
  action menu automatically (`list_checks()` returns sorted `_CHECKS`), and a direct emitter-power
  measurement is the textbook discriminator for a suspected laser-aging fault (the live `cap_case2`
  `recommended_action` already names exactly this measurement), so a competent proposer should select
  it. **This is the one item warranting live confirmation** (n≥1): verify the live controller proposes
  `laser_power_check` on case2 and still concludes laser. The deterministic spike proves the mechanism
  given the check runs; only live proves the LLM runs it. (This is the precise failure shape the prior
  investigation's case8 burned on — a fix that the real LLM's action/seed choices never let fire.)
- **Spurious abstention on the other cases?** Low. Each gold cause retains ≥1 affirmative check
  (table above), most via checks the live controller already runs (config_diff for case4/8,
  spatial_intensity for case3, detector_health for case6, tec_load/temp_correlation for case1/7). The
  gate can only *remove* positive links from nominal checks; it never removes negatives and never
  touches affirmative checks, so it cannot turn a genuinely-supported leader into an abstention.
- **Interaction with the deterministic sweeps.** The gate runs in `interpret_result` only; it does NOT
  touch the deterministic links added by `_finalize_conflicts` / `_promote_common_mode` /
  `_promote_change_cause` (those are minted directly in `loop.py`, never through interpret). So the M2/
  M3/M4 fixes are unaffected — case4 (change-cause promotion) and case8 (common-mode promotion) keep
  their affirmative deterministic links.
- **Residual M1 exposure.** The gate removes the *nominal-check over-crediting* path (the verified
  case5 mechanism). It does not constrain how the LLM weights a *genuinely affirmative* check — an
  over-weighted affirmative link could still saturate a hypothesis (capped at |LLR|≤2.0 per link and
  ±3.0 accumulated, so bounded, not eliminated). And the gate trusts the check's `affirmative`
  predicate: a check whose predicate is mis-specified would mis-gate. Both are narrower than the
  original M1 and are bounded by the existing log-odds clamps.
